                        archenroot • 2 years ago • edited

                        Hi Yegor. I am now facing following situation. While building event driven microservices I would like to monitor method execution. But to me I need method start and method finish as I would like to monitor later in some InfluxDB or ELK instances create alerts in case of some method in whole system takes too much time. Do I need 2 AspectJ annotations to implement or I can do this via single one?

                        Update
                        I realized myself according to Spring performance logging:


                        public class MyPerformanceMonitorInterceptor extends AbstractMonitoringInterceptor {
                             
                            public MyPerformanceMonitorInterceptor() {
                            }
                         
                            public MyPerformanceMonitorInterceptor(boolean useDynamicLogger) {
                                    setUseDynamicLogger(useDynamicLogger);
                            }
                         
                            @Override
                            protected Object invokeUnderTrace(MethodInvocation invocation, Log log) 
                              throws Throwable {
                                String name = createInvocationTraceName(invocation);
                                long start = System.currentTimeMillis();
                                log.info("Method " + name + " execution started at:" + new Date());
                                try {
                                    return invocation.proceed();
                                }
                                finally {
                                    long end = System.currentTimeMillis();
                                    long time = end - start;
                                    log.info("Method "+name+" execution lasted:"+time+" ms");
                                    log.info("Method "+name+" execution ended at:"+new Date());
                                     
                                    if (time > 10){
                                        log.warn("Method execution longer than 10 ms!");
                                    }            
                                }
                            }
                        }

                                                                                                                                                          Kapralov Sergey —

                                                                                                                                                          IMO cohesion is like elemental particle. Particles cannot be seen by microscope, they are usually discovered by observing the results of their collaboration with environment. Same with cohesion --- it can't be seen, observed and reasoned separately from coupling and reusability. That's why instead of staring to the object itself, guessing how many methods it should have in order to stay cohesive, one should check the clients who call them. Truly cohesive sets of methods are those, clients of which call (depend on) all of them, and there is no way of simplifying the client further and throwing any single method out of the set.

                                                                                                                                                          That's why `Books` interface will never be cohesive. Because doubtly ever you'll face a caller which will use *all* its methods in one place. That's why `Book` interface will never be cohesive --- because hardly all its clients will use all its methods at once. The main mistake of these interfaces is that they reflect the data tier. `Books` interface implicitly assumes some repository beneath, some database table, where you add book records. `Book` interface assumes that beneath there is some record, which should eventually be deleted. Depending on them is anti-OOP, anti-EO and anti-maintainable. They don't reflect and serve client needs, they reflect data tier beneath.

                                                                                                                                                          Nikolas Charalambidis —

                                                                                                                                                          Most of the pilots don't understand and don't even know how their aircraft works, let's say Airbus a380 with two full-length decks. A pilot needs to know how to operate it, to take a control when an exceptional behavior occurs and land safely. They really don't know the parts of the plain and are not able to change the behavior or fix the problem (even let's suppose they have a "time" for that).

                                                                                                                                                          Now here comes the analogy: let's say the world's biggest aircraft might be compared to the biggest Java framework, such as Spring Framework. I rely on the documentation and defined behavior. Although I am curious and interested to know its inner magic, explore it, however, it'd be only a tiny part of the entire framework I could understand in a reasonable time to be able to make changes in its source code. Developers are in charge of meeting deadlines and the frameworks are designed for that (the @RestController abstraction is self-explaining). We have no time to dig further and unluckily, it is not required since there is no direct business case.

                                                                                                                                                          According to your description, most of us are "bad developers", however, I feel that most of us (yes, me too) wish to be those "good developers" you described.
